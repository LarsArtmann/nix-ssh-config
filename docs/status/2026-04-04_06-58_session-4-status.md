### OpenSSH Post-Quantum Research (this session)

Researched current state of post-quantum SSH:

| Area                               | Status           | Details                                                                                               |
| ---------------------------------- | ---------------- | ----------------------------------------------------------------------------------------------------- |
| **Key exchange (PQ)**              | ✅ Done          | `mlkem768x25519-sha256` — ML-KEM hybrid, default since OpenSSH 10.0                                   |
| **Key exchange (PQ fallback)**     | ✅ Done          | `sntrup761x25519-sha512` — NTRU Prime hybrid, available since OpenSSH 8.5                             |
| **Authentication signatures (PQ)** | ❌ Not available | ML-DSA (CRYSTALS-Dilithium) not yet in OpenSSH. IETF draft exists (`ssh-mldsa44/65/87`). No timeline. |
| **Hybrid key types**               | ✅ KEX only      | All PQ algorithms in OpenSSH are hybrid (PQ + classical). Signature keys are classical only for now.  |

**Threat assessment:**

- **Key exchange**: Protected against "store now, decrypt later" — the urgent quantum threat. ✅
- **Authentication keys (Ed25519)**: Not quantum-safe, but an attacker needs both a captured handshake AND a future quantum computer to impersonate. Low urgency.

## g) Top #1 Question I Cannot Figure Out Myself

**Should the shared crypto constants module output strings or lists (or both)?**

The NixOS `services.openssh.settings` interface is inconsistent:

- `Ciphers` and `KexAlgorithms` accept **Nix lists** → rendered as multi-line sshd_config entries
- `MACs`, `HostKeyAlgorithms`, `PubkeyAcceptedAlgorithms` require **comma-separated strings**

Meanwhile, the Home Manager `programs.ssh.matchBlocks` `extraOptions` interface expects **strings** for everything.

So a shared module would need to output one of:

1. **Lists everywhere** — callers do `lib.concatStringsSep ","` where needed (current NixOS approach for some fields)
2. **Strings everywhere** — callers use raw (current HM approach)
3. **Both** — expose `pqKexList` and `pqKexString` variants (ugly but explicit)
4. **Lists + helper function** — expose lists + a `join = lib.concatStringsSep ","` that callers apply

This is a design decision that affects API ergonomics and I can't determine the "right" answer without knowing your preference for how the shared module should feel.
